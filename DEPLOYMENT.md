# JV Vault: Deployment & Collaboration Guide

To make the JV Vault portable across different systems (Mac, Linux, Windows) and ready for VPS hosting, follow these guidelines.

## 1. Quick Setup for Collaborators

If you've just cloned the repo, follow these steps to get running:

```bash
# 1. Install dependencies
npm install

# 2. Generate your unique session secret
npm run setup

# 3. Start development server
npm run dev
```

The database directory `data/` will be created automatically on your first run. The actual database files are ignored by Git to ensure your private vault data never leaves your machine.

---

## 2. Environment Variables

Create a `.env.local` file (or use `.env` for VPS) with the following:

- `SESSION_SECRET`: A 64-character hex string (generated by `npm run setup`).
- `VAULT_DB_PATH`: (Optional) Defaults to `./data/vault.db`. On a VPS, you might want an absolute path like `/var/lib/jv-vault/vault.db`.
- `NODE_ENV`: Set to `production` on your VPS.

---

## 3. VPS Deployment (Ubuntu/Linux)

### Prerequisites
- Node.js 20+
- `pm2` for process management (`npm install -g pm2`)

### Step-by-Step Deployment
1. **Clone and Build**:
   ```bash
   git clone <your-repo-url>
   cd jv-vault
   npm install
   npm run setup
   npm run build
   ```

2. **Run with PM2**:
   ```bash
   pm2 start npm --name "jv-vault" -- start
   ```

3. **Updating Data**:
   If you need to move your vault data to the VPS, copy only the `data/vault.db` file to the same location on the server. **Never commit `vault.db` to Git.**

---

## 4. Troubleshooting Windows Installation

If a collaborator on Windows encounters `node-gyp` or `Visual Studio` errors during `npm install`:

1.  **Check Node Version**: You are likely on a bleeding-edge version (like Node v25). **Downgrade to Node v20 or v22 (LTS)** using `nvm-windows`. This is the most common fix.
2.  **Install C++ Build Tools**: If you must stay on your current version, you need to install the "Desktop development with C++" workload via the Visual Studio Installer.

## 5. Sharing Data with Collaborators

The database file (`data/vault.db`) is **not** included in Git for security reasons. 

- When a collaborator first runs the app, they will see a **Setup Screen**. They will create their own private Master Password and start with a fresh, empty workspace.
- **To share your current data with them**: You must manually send them your `data/vault.db` file through a private channel. Once they place it in their `data/` folder, they can log in using **your** master password.

---

## 6. Key Security Rules
- **Encrypted Content**: All client and workspace data is encrypted at rest using AES-256-GCM. The key is derived from the Master Password you set during `/setup`.
- **Master Password**: This is the only thing that matters. If you lose it, the database is unrecoverable. It is **never** stored in the database.
- **Session Security**: The system uses database-backed session tokens. No manual secret configuration is required.
